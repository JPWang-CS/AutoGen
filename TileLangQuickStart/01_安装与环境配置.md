# 01 安装与环境配置

## 前置要求

### 硬件要求
- **华为昇腾NPU**：支持昇腾AI Core架构（如Ascend 910A/910B系列）

### 软件要求

| 要求 | 版本 |
|------|------|
| CANN | >= 8.3.RC1 |
| torch-npu | >= 2.6.0.RC1 |
| Python | >= 3.9 |

### NPU vs GPU 环境对比

| 项目 | NPU (tilelang-ascend) | GPU (上游 TileLang) |
|------|----------------------|---------------------|
| **硬件** | 华为昇腾 910A/910B | NVIDIA GPU (A100/H100等) |
| **计算框架** | CANN | CUDA |
| **PyTorch后端** | torch-npu (`x.npu()`) | PyTorch原生 (`x.cuda()`) |
| **源码仓库** | `tilelang-ascend` | `tilelang` (上游) |
| **安装脚本** | `bash install_ascend.sh` | `python setup.py install` |
| **编译器后端** | 毕昇编译器 → AscendC | NVCC → CUDA |
| **运行时** | aclRT | CUDA Runtime |

## 安装方式

### 第一步：CANN环境准备

假设您已搭建好包含CANN和torch-npu的昇腾环境，首先设置CANN环境变量：

```bash
source {your-cann-installed-path}/ascend-toolkit/set_env.sh
```

### 第二步：源码编译安装 TileLang-Ascend

```bash
# a) 下载源码
git clone --recursive https://github.com/tile-ai/tilelang-ascend.git
cd tilelang-ascend

# b) 编译安装
bash install_ascend.sh

# c) 设置环境变量
source set_env.sh
```

### 验证安装

```python
import tilelang
print(f"TileLang 版本: {tilelang.__version__}")
```

运行官方示例来验证完整环境：

```bash
cd examples/gemm
python example_gemm.py
```

成功后会打印：

```
init successful!
Kernel Output Match!
```

import tilelang

tilelang.disable_cache()

## 第一个 TileLang-Ascend 程序

下面是一个运行在昇腾NPU上的向量加法示例：

```python
import tilelang
import tilelang.language as T
import torch

@tilelang.jit(out_idx=[-1])
def vec_add(M, N, block_M, block_N, dtype="float"):
    m_num = M // block_M
    n_num = N // block_N
    VEC_NUM = 2  # 每个AI Core有2个Vector计算单元

    @T.prim_func
    def main(
        A: T.Tensor((M, N), dtype),
        B: T.Tensor((M, N), dtype),
        C: T.Tensor((M, N), dtype),
    ):
        with T.Kernel(m_num * n_num, is_npu=True) as (cid, vid):
            bx = cid // n_num
            by = cid % n_num

            # 分配Unified Buffer（Vector Core的片上存储）
            a_ub = T.alloc_ub((block_M // VEC_NUM, block_N), dtype)
            b_ub = T.alloc_ub((block_M // VEC_NUM, block_N), dtype)
            c_ub = T.alloc_ub((block_M // VEC_NUM, block_N), dtype)

            # 在Vector Core作用域内执行数据搬运和计算
            with T.Scope("V"):
                # GM -> UB：数据加载
                T.copy(A[bx * block_M + vid * block_M // VEC_NUM, by * block_N], a_ub)
                T.copy(B[bx * block_M + vid * block_M // VEC_NUM, by * block_N], b_ub)

                T.barrier_all()
                # 向量计算
                T.tile.add(c_ub, a_ub, b_ub)
                T.barrier_all()

                # UB -> GM：数据写回
                T.copy(c_ub, C[bx * block_M + vid * block_M // VEC_NUM, by * block_N])

    return main


# 运行
func = vec_add(1024, 1024, 128, 256)

torch.manual_seed(0)
a = torch.randn(1024, 1024).npu()
b = torch.randn(1024, 1024).npu()

print("init successful!")

c = func(a, b)
ref_c = a + b

torch.testing.assert_close(c, ref_c, rtol=1e-2, atol=1e-2)
print("Kernel Output Match!")
```

### 关键概念说明

- **`is_npu=True`**：指定在NPU上运行，返回 `(cid, vid)` 元组
  - `cid`：计算任务编号（对应逻辑核）
  - `vid`：Vector计算单元索引（0或1）
- **`T.alloc_ub()`**：分配Unified Buffer，Vector Core的片上存储
- **`T.Scope("V")`**：Vector Core作用域，数据搬运和向量计算需在此作用域内
- **`T.tile.add()`**：Vector Core的向量加法指令
- **`.npu()`**：将PyTorch张量放置到NPU设备上

### NPU vs GPU 向量加法对比

相同功能的向量加法，NPU 和 GPU 的写法差异：

**NPU (tilelang-ascend)**:
```python
# NPU: is_npu=True, T.alloc_ub, T.Scope("V"), T.tile.add
with T.Kernel(num_tasks, is_npu=True) as (cid, vid):
    a_ub = T.alloc_ub((block_M // 2, block_N), dtype)   # UB 存储
    b_ub = T.alloc_ub((block_M // 2, block_N), dtype)
    c_ub = T.alloc_ub((block_M // 2, block_N), dtype)
    with T.Scope("V"):                                   # Vector Core 域
        T.copy(A[...], a_ub)
        T.copy(B[...], b_ub)
        T.tile.add(c_ub, a_ub, b_ub)                    # 向量指令
        T.copy(c_ub, C[...])
```

**GPU (上游 TileLang)**:
```python
# GPU: threads=N, T.alloc_shared, T.Parallel, 标量运算
with T.Kernel(num_tasks, threads=128) as (bx):
    a_shared = T.alloc_shared((block_M, block_N), dtype)  # Shared Memory
    b_shared = T.alloc_shared((block_M, block_N), dtype)
    T.copy(A[...], a_shared)
    T.copy(B[...], b_shared)
    for i, j in T.Parallel(block_M, block_N):             # 线程级并行
        C[bx * block_M + i, by * block_N + j] = a_shared[i, j] + b_shared[i, j]
```

**核心差异**:

| 差异点 | NPU | GPU |
|--------|-----|-----|
| Kernel 启动 | `is_npu=True`, 返回 `(cid, vid)` | `threads=N`, 返回 `(bx,)` |
| 片上存储 | `T.alloc_ub()` (UB) | `T.alloc_shared()` (Shared Memory) |
| 计算域 | `T.Scope("V")` 显式声明 | 无需，自动管理 |
| 向量计算 | `T.tile.add()` 批量指令 | `T.Parallel` + 标量 `+` |
| 数据放置 | `.npu()` | `.cuda()` |

## TileLang编译和运行流程

TileLang Ascend的工作流程分为编译和运行两个阶段：

**编译阶段**：
1. 多轮lower转换：TileLang算子代码根据NPU硬件特性进行多级降级，生成针对昇腾硬件优化的TensorIR表示
2. AscendC代码生成：基于TensorIR，使用专门的Ascend Codegen模块生成对应的AscendC代码
3. 动态库编译：通过毕昇编译器（bisheng）将AscendC代码编译成动态链接库（.so文件）

**运行阶段**：
1. 库文件加载：通过ctypes将.so文件加载到Python环境中
2. 函数封装：将算子封装为可调用的Python函数对象
3. 执行调用：用户以普通Python函数方式调用，提供输入张量即可在昇腾NPU上执行计算并获得结果

## 常见问题

### Q: 编译时间较长？
首次编译需要生成和优化AscendC代码，后续调用使用缓存会快很多。

### Q: 如何查看生成的内核代码？
```python
print(func.get_kernel_source())
```

### Q: 数据精度对比？
TileLang-Ascend的GEMM等计算使用float16输入和float32累加，最终精度对比使用：
```python
torch.testing.assert_close(c, ref_c, rtol=1e-2, atol=1e-2)
```
