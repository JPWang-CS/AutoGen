# 02 语言基础

## 编程模型概述

TileLang-Ascend 采用 **基于 Tile 的编程模型**，开发者通过定义 **Tile（数据块）** 级别的操作来编写高性能NPU内核。核心思路是：

1. 将全局数据划分为多个 Tile
2. 每个AI Core负责处理一个或多个 Tile
3. 利用昇腾NPU的多级内存层次进行高效计算

### 昇腾NPU内存层次

```
┌─────────────────────────────────────────────────────────┐
│                    Global Memory (GM)                    │  大容量、高延迟
│          通过 T.copy 访问，所有AI Core共享                  │
├───────────────────────┬─────────────────────────────────┤
│   L1 Buffer (Cube)    │  Unified Buffer / UB (Vector)   │  片上高速存储
│  Cube Core的缓存       │  Vector Core的计算缓冲区          │
│  T.alloc_L1()         │  T.alloc_ub()                   │
├───────────────────────┴─────────────────────────────────┤
│  L0A / L0B (矩阵输入寄存器)  →  L0C (矩阵输出寄存器)        │  寄存器级存储
│  T.alloc_L0A() / T.alloc_L0B()    T.alloc_L0C()         │
└─────────────────────────────────────────────────────────┘
```

### Developer模式 vs Expert模式

TileLang-Ascend提供两种编程接口：

**Developer模式（推荐入门使用）**：
- `T.alloc_shared()` → 编译器自动映射到L1/UB
- `T.alloc_fragment()` → 编译器自动映射到L0C
- 无需关心底层存储是L1还是UB，编译器自动分析

**Expert模式（高级优化）**：
- `T.alloc_L1()`、`T.alloc_ub()`、`T.alloc_L0A()`、`T.alloc_L0B()`、`T.alloc_L0C()`
- 显式控制每个buffer的存储位置
- 可进行更精细的性能优化

### NPU vs GPU 内存层次对比

```
NPU (昇腾 AI Core)                     GPU (NVIDIA SM)
┌──────────────────────┐               ┌──────────────────────┐
│  Global Memory (GM)   │               │  Global Memory        │
│  大容量、高延迟        │               │  (HBM) 大容量高延迟    │
├──────────────────────┤               ├──────────────────────┤
│  L1 Buffer (Cube)     │               │  Shared Memory        │
│  UB (Vector)          │               │  所有线程共享           │
│  片上高速存储          │               │  片上高速存储           │
├──────────────────────┤               ├──────────────────────┤
│  L0A/L0B → L0C        │               │  Registers            │
│  矩阵输入 → 矩阵输出    │               │  线程私有寄存器         │
│  (Cube Core寄存器)     │               │                      │
└──────────────────────┘               └──────────────────────┘
```

| 层级 | NPU | GPU |
|------|-----|-----|
| 全局存储 | GM (Global Memory) | HBM (Global Memory) |
| 片上存储 | L1 (Cube) / UB (Vector) | Shared Memory |
| 寄存器 | L0A/L0B/L0C (Cube) | Registers (每线程) |
| 分配方式 | `T.alloc_L1/ub/L0C` | `T.alloc_shared/fragment` |
| 关键区别 | **Cube与Vector分离**，两种核各有独立存储 | 统一的 Shared Memory，所有线程共享 |

## 内核定义：`@T.prim_func`

所有 TileLang 内核都通过 `@T.prim_func` 装饰器定义：

```python
import tilelang.language as T

@T.prim_func
def my_kernel(
    A: T.Tensor((M, K), "float16"),  # 输入张量 A
    B: T.Tensor((K, N), "float16"),  # 输入张量 B
    C: T.Tensor((M, N), "float16"),  # 输出张量 C
):
    # 内核逻辑
    ...
```

**接口说明**：
- `T.Tensor(shape, dtype)`：声明张量参数，指定形状和数据类型
- 函数体内的代码会被TileLang编译器转换为可在NPU上运行的AscendC代码
- 张量形状可以使用外部变量（如 `M`、`N`、`K`），通过闭包传入

### 动态符号维度

TileLang 支持两种方式引入动态（符号化）维度：

**方式一：注解级符号 T.dyn**
```python
K = T.dyn['K']  # dtype 默认为 int32
@T.prim_func
def foo(A: T.Tensor((K,), 'float32')):
    N = A.shape[0]  # 在内核体内通过 shape 读取绑定的尺寸
    for i in T.serial(N):
        ...
```

**方式二：显式 Var 符号 T.dynamic**
```python
K = T.dynamic('K', 'int32')  # 或 T.dynamic('K')，默认 int32
@T.prim_func
def bar(A: T.Tensor((K,), 'float32')):
    for i in T.serial(K):
        ...
```

> **注意**：`T.symbolic(name, dtype)` 是 `T.dynamic` 的已弃用别名，请优先使用 `T.dynamic`。

## 内核入口：`T.Kernel`

`T.Kernel` 定义了NPU内核的执行上下文，创建数据Tile与逻辑核的绑定关系：

```python
with T.Kernel(num_tasks, is_npu=True) as (cid, vid):
    # cid: 计算任务编号
    # vid: Vector计算单元索引（0或1）
    ...
```

### 接口参数

| 参数 | 类型 | 说明 |
|------|------|------|
| `num_tasks` | int | 并发任务数量（通常为数据分块总数） |
| `is_npu` | bool | **必须设为 True**，指定在NPU上运行 |

### 返回值

返回元组 `(cid, vid)`：
- `cid`：计算任务编号，范围为 `[0, num_tasks)`，每个Tile对应一个计算任务
- `vid`：Vector计算单元索引，值为0或1。因为每个AI Core有两个Vector计算单元（VEC_NUM=2）

### 使用示例

```python
M, N = 1024, 1024
block_M, block_N = 128, 128
m_num = M // block_M
n_num = N // block_N

@T.prim_func
def main(A: T.Tensor((M, N), dtype), B: T.Tensor((M, N), dtype), C: T.Tensor((M, N), dtype)):
    with T.Kernel(m_num * n_num, is_npu=True) as (cid, vid):
        bx = cid // n_num  # 行方向block索引
        by = cid % n_num   # 列方向block索引
        ...
```

### Cube Core vs Vector Core 的 Kernel 使用

**纯Cube操作（如GEMM）**：`vid` 不使用，用 `_` 忽略：
```python
with T.Kernel(m_num * n_num, is_npu=True) as (cid, _):
    ...
```

**Vector操作**：使用 `vid` 进行子Tile切分：
```python
VEC_NUM = 2
with T.Kernel(m_num * n_num, is_npu=True) as (cid, vid):
    # vid=0 处理上半部分，vid=1 处理下半部分
    a_ub = T.alloc_ub((block_M // VEC_NUM, block_N), dtype)
    T.copy(A[bx * block_M + vid * block_M // VEC_NUM, by * block_N], a_ub)
    ...
```

### NPU vs GPU Kernel 启动对比

**NPU (tilelang-ascend)**:
```python
# Cube操作: 返回 (cid, _)  cid=逻辑核编号
with T.Kernel(m_num * n_num, is_npu=True) as (cid, _):
    bx = cid // n_num
    by = cid % n_num

# Vector操作: 返回 (cid, vid)  vid=Vector单元索引(0或1)
VEC_NUM = 2
with T.Kernel(m_num * n_num, is_npu=True) as (cid, vid):
    # vid=0 和 vid=1 分别处理一个子Tile
```

**GPU (上游 TileLang)**:
```python
# GPU: 指定 threads 数量，返回线程块索引
with T.Kernel(num_blocks, threads=128) as (bx):
    # bx = 线程块编号
    # 线程内通过 threadIdx 获取线程索引
```

| 差异点 | NPU | GPU |
|--------|-----|-----|
| 启动参数 | `is_npu=True` | `threads=N` |
| 返回值 | `(cid, vid)` 元组 | `(block_idx,)` |
| 并行粒度 | AI Core 级别 | 线程块级别 |
| 子单元 | `vid` 对应 2 个 Vector 单元 | `threadIdx` 对应数百个线程 |
| 无线程概念 | 一个 AI Core 处理一个 Tile | 一个线程块内多线程协作 |

## 作用域：T.Scope

在昇腾NPU上，不同计算引擎需要在对应的作用域内操作：

```python
# Cube Core 作用域（矩阵乘法相关操作）
with T.Scope("C"):
    T.copy(A[bx * block_M, k * K_L1], A_L1)  # GM -> L1
    T.gemm_v0(A_L1, B_L1, C_L0, init=True)    # 矩阵乘法
    T.copy(C_L0, C[bx * block_M, by * block_N])  # L0C -> GM

# Vector Core 作用域（向量计算相关操作）
with T.Scope("V"):
    T.copy(A[bx * block_M + vid * block_M // VEC_NUM, by * block_N], a_ub)  # GM -> UB
    T.tile.add(c_ub, a_ub, b_ub)  # 向量加法
    T.copy(c_ub, C[bx * block_M + vid * block_M // VEC_NUM, by * block_N])  # UB -> GM
```

**规则总结**：
- `T.Scope("C")`：Cube Core作用域，包含L1/L0A/L0B/L0C的操作
- `T.Scope("V")`：Vector Core作用域，包含UB的搬运和向量计算
- `T.copy` 的 GM↔L1 操作可在 `T.Scope("C")` 内
- `T.copy` 的 GM↔UB 操作可在 `T.Scope("V")` 内
- `T.tile.*` 向量指令必须在 `T.Scope("V")` 内

> **GPU对比**：GPU 上没有 `T.Scope` 概念，Shared Memory 和计算统一由线程管理。NPU 因为 Cube/Vector 双核架构，需要显式声明操作域。

## 内存分配

### Developer模式

#### T.alloc_shared — 分配片上共享存储

```python
A_shared = T.alloc_shared((BM, BK), "float16")
```

编译器会根据上下文自动将 `T.alloc_shared` 映射到 L1 Buffer（Cube操作用）或 Unified Buffer（Vector操作用），开发者无需显式指定。

#### T.alloc_fragment — 分配Fragment存储

```python
C_frag = T.alloc_fragment((BM, BN), "float32")
```

编译器会根据上下文自动映射到 L0C（Cube矩阵乘输出）或其他寄存器级存储。

### Expert模式

#### T.alloc_L1 — 分配L1 Buffer

```python
A_L1 = T.alloc_L1((block_M, K_L1), "float16")
```

L1 Buffer是Cube Core的高速缓存，用于暂存矩阵乘法的输入数据。

#### T.alloc_ub — 分配Unified Buffer

```python
a_ub = T.alloc_ub((block_M // VEC_NUM, block_N), "float32")
```

Unified Buffer是Vector Core的计算缓冲区，所有向量操作（T.tile.*）都在UB上进行。

**VEC_NUM说明**：每个AI Core有2个Vector计算单元（VEC_NUM=2），因此需要将Tile按VEC_NUM均分，每个Vector单元处理一个子Tile。

#### T.alloc_L0A / T.alloc_L0B — 分配矩阵输入寄存器

```python
A_L0 = T.alloc_L0A((block_M, block_K), "float16")
B_L0 = T.alloc_L0B((block_K, block_N), "float16")
```

L0A/L0B是Cube Core的矩阵输入寄存器，用于存放矩阵乘法的左右输入矩阵。

#### T.alloc_L0C — 分配矩阵输出寄存器

```python
C_L0 = T.alloc_L0C((block_M, block_N), "float32")
```

L0C是Cube Core的矩阵输出寄存器，用于存放矩阵乘法的累加结果。

### T.alloc_var — 分配局部变量

```python
flag = T.alloc_var("bool", init=False)
counter = T.alloc_var("int32", init=1)
```

用于分配标量变量，适用于条件标志位、循环计数器等场景。

## JIT 编译与执行：@tilelang.jit

`@tilelang.jit` 是最常用的编译执行方式，触发即时编译：

```python
@tilelang.jit(out_idx=[-1])
def my_kernel_func(M, N, K, block_M, block_N, dtype="float16"):
    @T.prim_func
    def kernel(
        A: T.Tensor((M, K), dtype),
        B: T.Tensor((K, N), dtype),
        C: T.Tensor((M, N), dtype),
    ):
        ...
    return kernel
```

### 接口参数

| 参数 | 类型 | 说明 |
|------|------|------|
| `out_idx` | list[int] | 输出张量在参数列表中的索引，`[-1]` 表示最后一个参数 |
| `pass_configs` | dict | 编译优化配置（详见06_自动调优.md） |
| `target` | str | 编译目标，默认为NPU |
| `workspace_idx` | list[int] | workspace张量的参数索引列表 |

### 调用方式

```python
# 编译时只传配置参数
kernel = my_kernel_func(M=1024, N=1024, K=1024, block_M=128, block_N=256)

# 运行时传入NPU张量
import torch
a = torch.randn(1024, 1024).half().npu()  # float16输入
b = torch.randn(1024, 1024).half().npu()
c = kernel(a, b)  # 自动执行并返回结果
```

### 数据准备

```python
# float16 数据
a = torch.randn(M, K).half().npu()

# float32 数据
a = torch.randn(M, K).npu()

# 精度对比
torch.testing.assert_close(c, ref_c, rtol=1e-2, atol=1e-2)
```

## 完整示例：向量加法内核

```python
import tilelang
import tilelang.language as T
import torch

@tilelang.jit(out_idx=[-1])
def vec_add(M, N, block_M, block_N, dtype="float"):
    m_num = M // block_M
    n_num = N // block_N
    VEC_NUM = 2

    @T.prim_func
    def main(
        A: T.Tensor((M, N), dtype),
        B: T.Tensor((M, N), dtype),
        C: T.Tensor((M, N), dtype),
    ):
        with T.Kernel(m_num * n_num, is_npu=True) as (cid, vid):
            bx = cid // n_num
            by = cid % n_num

            a_ub = T.alloc_ub((block_M // VEC_NUM, block_N), dtype)
            b_ub = T.alloc_ub((block_M // VEC_NUM, block_N), dtype)
            c_ub = T.alloc_ub((block_M // VEC_NUM, block_N), dtype)

            with T.Scope("V"):
                T.copy(A[bx * block_M + vid * block_M // VEC_NUM, by * block_N], a_ub)
                T.copy(B[bx * block_M + vid * block_M // VEC_NUM, by * block_N], b_ub)

                T.barrier_all()
                T.tile.add(c_ub, a_ub, b_ub)
                T.barrier_all()

                T.copy(c_ub, C[bx * block_M + vid * block_M // VEC_NUM, by * block_N])

    return main

# 编译并运行
func = vec_add(1024, 1024, 128, 256)
a = torch.randn(1024, 1024).npu()
b = torch.randn(1024, 1024).npu()
c = func(a, b)
torch.testing.assert_close(c, a + b, rtol=1e-2, atol=1e-2)
print("Kernel Output Match!")
```

## @T.macro — 可复用代码块

TileLang提供 `@T.macro` 用于定义可复用的代码片段，编译时内联展开：

```python
@T.macro
def init_flag():
    T.set_flag("mte1", "mte2", 0)
    T.set_flag("m", "mte1", 0)

# 在Kernel中使用
with T.Kernel(core_num, is_npu=True) as (cid, _):
    with T.Scope("C"):
        init_flag()
        ...
```
